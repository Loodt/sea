## Question Selection
Rank by: information gain > priority > feasibility > domain data density > staleness > dependency-unlocking.
**Pruning mode** (auto when open >15 OR open:resolved >2:1): prioritize kill-check/synthesis, deprioritize mechanism, prefer existing questions over new landscape scope.

## Question Types
| Type | Cap | Selection guidance |
|------|-----|-------------------|
| landscape | 5 | Broad survey. Dispatch first to establish frontier. |
| kill-check | 5 | Hypothesis falsification. Prefer when >3 open pathways. |
| data-hunt | 3 | Specific values. High exhaustion risk. |
| mechanism | 5 | How/why. Full-budget convergence normal. |
| synthesis | 2 | Combine store findings. Dispatch opportunistically. |

Caps enforced via `QUESTION_TYPE_ITERATION_CAP` (types.ts). Selection MUST output `questionType`.

## Expert Convergence
- **answered** — resolved with well-evidenced findings
- **killed** — non-viable (equally valuable as answered)
- **narrowed** — partial progress. Re-dispatch MUST include prior handoff findings *(debt #6)*
- **exhausted** — diminishing returns. Integration MUST create `[DERIVED: exhaustive-search]` finding
- **crashed** — all iterations failed (infrastructure). NOT exhausted. Re-dispatch eligible

## Expert Library
Personas stored by hash in expert-library/library.jsonl. Utility = avgIG x log(dispatches + 1). Factory checks library for matches by questionType + domain overlap; adapts high-scoring persona (>2.0) instead of creating fresh.

## Finding Graduation
Provisional findings auto-graduate to verified when: confidence >= 0.85, tag = SOURCE with URL, age >= 3 iterations, not contradicted. Runs after each dispatch.

## Step Gates
- **Crash gate:** Step exits non-zero + <100 bytes → crash recovery, skip downstream *(debt #1)*
- **Crash-score exclusion:** Crash scores MUST NOT trigger rollback *(debt #2)*
- **Summarize completeness:** Persisted findings must match produced. Log PERSISTENCE_GAP if not
- **Summarize before synthesize:** Store current before report. Finding IDs MUST exist in findings.jsonl
- **Score persistence:** Evaluate writes JSON scores block. Loop parses/persists. Evaluate MUST NOT write scores.jsonl

## Running the Conductor
When running `sea conduct` or any long-running CLI command, launch it as a background task and **wait for the completion notification**. Do NOT poll the output file or check progress repeatedly. Each inner loop step spawns a full Claude Code subprocess that can take minutes. Trust the process and report results only after the command finishes.

## Hard Rules
- Context budgets per agent: enforced in `types.ts` CONTEXT_BUDGETS
- Persona max 60 lines — consolidate before adding
- Knowledge store is source of truth — not output/ reports; summary.md max 2KB
- Tag every claim: `[SOURCE: url]` `[DERIVED: method]` `[ESTIMATED: basis]` `[ASSUMED]` `[UNKNOWN]`
- `[UNKNOWN]` over untagged guess; anchor comparisons (baseline, magnitude, conditions)
- Evaluate agent is independent critic — never sees persona or goal
- One evolution change at a time — hypothesis + measurement + rollback trigger
- Rollback-first on >15% score drop from 3-iter rolling average (excluding crashes)
- Min 2 iterations before judging; check failure-patterns/ before proposing changes
- Evolve: 2 heuristic failures on same issue → classify as infrastructure, stop trying
- Summarize MUST receive FULL raw findings — never truncate (32KB budget)
- ALL failure patterns loaded — no subset limits
- Expert store writes idempotent: F9XX IDs → sequential; integration deduplicates by claim text
- New generalizable failures → failure-patterns/; new successes → success-patterns/

## Evolution Protocol
Three valid outcomes:
1. **Behavioral change** — modify persona. Requires: hypothesis, measurement, rollback trigger.
2. **Strategic advancement** — update targets when questions resolved/blocked.
3. **No-change hold** — declare working. Record lineage with reasoning.

**Stagnation** (2 consecutive non-crash iters, zero findings + zero resolved): classify → exhausted (pivot question), blocked (mark, pivot), wrong (kill, reformulate).

Experts are disposable specialists — knowledge store is persistent memory, not persona.

## Infrastructure Debt
Open gaps between protocol and implementation. Do not attempt heuristic fixes — each requires code.

1. **Crash gate in loop.ts** — `runStep()` logs warning but continues. Fix: skip downstream on crash, set crash flag.
2. **Crash-score exclusion** — Score type lacks `crashIteration` field. Fix: add field, filter in `isRegressing()`.
3. **Summarize completeness gate** — No post-summarize findings count check.
4. **Early exhaustion mid-loop** — Expert loop runs all maxIterations regardless of findings. Mitigated by type caps. Fix: zero findings after iter 2 → cap remaining to 1.
5. **Exhaustion-as-finding (partial)** — Prompt-level exists. Code-level synthetic finding would be more robust.
6. **Narrowed dispatch rebinding** — Re-dispatched narrowed questions lose prior context. Fix: inject prior handoff into expert creation.
7. **Finding duplication** — Expert + integration both write findings.jsonl. Fix: integration deduplicates by claim text.

## Validated Dispatch Patterns
See `dispatch-patterns.md` for full observations. Key behavioral rules:
- Domain-specific experts converge faster — preserve emphasis on "defining trait" + "core values"
- Kill signals prune entire branches — never deprioritize kill-check questions
- Questions decomposing to one-expert-one-domain dispatch most efficiently
- Knowledge accumulation compounds: ~5x efficiency gain after ~40-50 findings in store
- Synthesis dispatches cheapest (1-2 iters) — generate opportunistically when store is rich
- Data-rich domains yield 5-10x more findings/iter than data-sparse
- Exhaustion is bimodal: substantial findings or nothing — no partial exhaustion
